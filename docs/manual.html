1. はじめに / 全体像

みんなの出勤簿 は、ブラウザで打刻 → 期間ごとの実働集計 → 社労士提出用の出勤簿 xlsx 生成 までを一つにまとめた PWA（プログレッシブ Web アプリ）です。Firebase の無料プランで稼働しています。

主な機能

  • 打刻（出勤・退勤・休憩・休憩終わり）：ブラウザの大型ボタンで記録
  • 本日の打刻一覧：誰が今勤務中／休憩中／退勤済みかを一目で確認
  • 期間集計：月2回制（1〜15日 / 16〜末）の出勤マトリクス表示・CSV 出力
  • 打刻管理：打刻漏れの追加・修正・削除（履歴を editLogs に残す）
  • 社員マスタ：氏名・部門・社員番号の管理
  • 社労士書類生成：1か月分の出勤簿 xlsx を自動生成（残業・深夜・週40h アラート付き）
  • 設定：自分の表示名変更・ユーザー権限の管理

2. ロール（権限）

権限は3層あります。最初に新規登録した人が自動的に office に昇格し、それ以降は全員 staff として登録されます。office が必要に応じて manager や office に昇格させてください。

3. 初回セットアップ

  1. URL を開く：https://attendance-app-b047c.web.app
  2. 新規登録：「新規登録はこちら」→ 表示名・メール・パスワードを入力 →「登録」
  3. 初回登録者は自動的に office になり、すぐ全機能にアクセス可能
  4. 社員マスタの確認：/employees で 98 名のシード（上野／谷川…）が入っているか確認
  5. 他のスタッフを招待：各自に「/register で登録してください」と案内
  6. 必要に応じて /settings でロールを manager や office に昇格
💡 キオスク端末（事務所共有 PC など）で打刻させたい場合は、その端末用の専用アカウントを作って staff でログインしっぱなしにします。打刻時は氏名選択 + 出勤ボタンの2タップでOK。

4. ログイン / 新規登録

ログイン（/login）

  1. メールアドレスとパスワードを入力
  2. 「ログイン」をタップ
  3. 成功すれば自動的に /punch に遷移

新規登録（/register）

  1. 表示名（田中太郎、〇〇店など）
  2. メールアドレス
  3. パスワード（6文字以上）と確認入力
  4. 「登録」をタップ
✓ 登録は誰でも可能ですが、初期権限は staff です。社内利用を前提にしているため、URL は社外に出さないでください。

7. 期間集計（/period）

月2回制（H1: 1〜15日 / H2: 16〜末日）の集計画面。社員 × 日付のマトリクスで一目で把握できます。

画面の操作

  • ← 前期間 / 次期間 →：期間を移動（URL の ?period=2026-04-H1 で固定可）
  • 今期間：現在の期間にジャンプ
  • ⇩ CSV出力：表示中の期間を CSV ダウンロード（社員ID／氏名／部門／日付／曜日／出勤／退勤／休憩h／実働h／備考）

サマリカード

  • 対象社員：active な社員数
  • 総打刻数：期間内の打刻件数
  • 未退勤(要対応)：出勤打刻のみで退勤がない日（黄色セル）
  • 合計実働(簡易)：期間内の全社員実働時間合計

マトリクスの色分け

  • 緑：通常勤務（出勤＋退勤）
  • 黄：打刻漏れ（出のみ／退のみ）— /admin で要修正
  • 薄赤：日曜日
  • 薄青：土曜日
  • —：未出勤

8. 打刻管理（/admin） manager office

打刻漏れの補正、誤打刻の修正、削除を行います。すべての操作は editLogs に履歴として残ります。

使い方

  1. 社員と日付を選択
  2. その日の打刻一覧が表示される
  3. 編集：行右の「編集」 → モーダルで種別・日時・理由を入力 → 保存
  4. 削除：行右の「削除」 → 理由を入力 → 確認
  5. 追加：「+ 打刻を追加」ボタン → 新しい打刻をモーダルで入力
⚠ 修正・削除には 必ず理由 が必要です。これは法的監査に対応するための措置で、後から「何のために変えたのか」を追跡できるようにしています。

修正履歴の確認

ページ下部「修正履歴（editLogs）最新30件」を展開すると、誰が・いつ・何を・なぜ変更したかが見られます。

9. 社員マスタ（/employees） office

追加

  1. 上部「新規追加」セクションに No.・氏名・部門・社員番号を入力
  2. 「追加」 → E099 のような連番 ID で登録される

編集

一覧の入力欄に直接タイプして編集 → フォーカスを外すと自動保存。

退職処理

「在籍」のチェックを外す → 打刻画面のドロップダウンから消える。打刻履歴は残るので過去データは無傷。

完全削除

右端の「削除」 → 確認後に employees から消える。打刻履歴は残ります（punches ツリーは独立）。

10. 社労士書類生成（/attendance） office

1か月分の出勤簿を、残業・深夜・週40h 集計付きで xlsx にまとめる画面。打刻データから自動生成するため、別途 Excel ファイルをアップロードする必要はありません。

生成手順

  1. 「① 対象月を選んで生成」で対象月を選択（既定: 当月）
  2. 「📊 生成」をタップ
  3. /punches から自動集計 → 「② 過去の出勤簿」リストに追加 + サマリ表示
  4. サマリ表で人ごとの労働時間・残業・深夜・アラートを確認
  5. 必要なら個人行右の「詳細」で日別の打刻と編集
  6. 右上「全員分ダウンロード (xlsx)」 → 1ファイル（複数シート）でダウンロード

ダウンロード xlsx の構成

1ファイルに複数のシートが入っています。Excel/Numbers 下部のシートタブで切り替えて閲覧してください。

  • サマリシート：作業番号・氏名・部門・総労働時間・総残業時間・総深夜時間・アラート（全員1行ずつ）
  • E001_上野、E002_谷川 ……：1人につき1シート。日別の打刻と労働時間、上部に集計アラート行

サマリのアラート列

過去の出勤簿の再表示

「② 過去の出勤簿」リストから過去月をクリックすると、保存済みデータが表示されます。再生成は不要。

生成済み月の上書き

同じ月で再度「生成」を押すと、確認ダイアログ「再生成して上書きしますか？」が出ます。OK で最新の /punches から再集計されます（/admin の手動修正もそのまま反映）。

12. 実働時間の計算ルール

すべての画面で共通に使われている computeWorkHours 関数のロジックを説明します。

基本式

実働時間 = (退勤 − 出勤) − Σ休憩時間

1日に複数の出勤／退勤がある場合（テレワーク再出勤など）は、各セッションごとに上記を算出して合計します。

状態走査アルゴリズム

打刻を時系列順に並べ、以下のルールで状態を更新しながら「勤務中」区間の合計を加算します。

現在状態打刻処理新状態
none / doneinworkStart 記録working
workingoutworkMs += (now − workStart) / sessions++done
workingbreak_startworkMs += (now − workStart)on_break
on_breakbreak_endbreakMs += (now − breakStart)working
on_breakout未閉休憩は計上せずセッション終了done

残業 / 深夜の判定

  • 日残業: その日の労働時間 − 法定 8h（負ならゼロ）
  • 週残業: 月曜起点の週内累計が 40h を超えた分
  • 月残業: 月内合計が 45h（36協定上限）を超えるとアラート
  • 深夜労働: 22:00〜翌5:00 に該当する時間（前日 0:00 越し勤務にも対応）

休憩の扱い

旧仕様の「一律 0.75h（45分）控除」は廃止。break_start と break_end 打刻ペアから実時間を計上します。

💡 6時間未満勤務でも休憩を記録できます（労基法上は6時間未満は休憩義務なしですが、自由意思での記録は妨げません）。

未閉ペアの扱い

  • 退勤打刻なし: その日の労働時間は「不明」扱い（黄色アラート）
  • 休憩終わり打刻なし: その休憩は計上せず、退勤時にセッション終了
  • いずれも /admin から手動補正可能（履歴は editLogs に残る）

14. よくある質問・トラブル対応

Q. 打刻ボタンが押せない（グレーアウト）

現在の状態と矛盾する打刻はできない仕様です。たとえば「勤務中」のときは出勤ボタンは押せません。状態を確認して、適切なボタンを押してください。

Q. 休憩終わりを押し忘れて退勤してしまった

その休憩は計上されません（未閉ペア）。/admin（manager / office）から該当日に break_end 打刻を追加してください。

Q. 退勤後にテレワークで仕事を再開したい

もう一度「出勤」ボタンを押せば、新しいセッションが始まります。退勤後の移動・休憩時間は労働時間に含まれません。

Q. 全員分ダウンロードしたファイルにアラートが出ない

アラートはサマリシートの「アラート」列と、各人シート上部の「⚠」行に出ます。Excel/Numbers 下部のシートタブで切り替えて確認してください。

Q. 過去の月のデータを再計算したい

/attendance でその月を選んで「📊 生成」を押すと「再生成して上書きしますか？」と聞かれます。OK で最新の打刻状態（/admin での手動補正含む）から再集計されます。

Q. 自分の表示名を変えたい

/settings の「表示名」欄を変えて「保存」。即時反映されます。

Q. 同じメールで二重登録できますか？

いいえ。1メール1アカウントです。ログインで使ってください。

Q. パスワードを忘れた

現バージョンにはパスワードリセット機能がありません。office に Firebase コンソールでパスワードリセットメールを送ってもらってください。

Q. 打刻データが他人に見えるのは問題ない？

社内利用前提のため、認証済みユーザーは全員の打刻状況を閲覧できます。これは「誰がいつ来てるか」を全員で見える化するための仕様です。外部公開は禁止。

Q. オフラインでも打刻できる？

打刻には認証＋RTDB通信が必要です。完全オフライン状態では失敗します。Wi-Fi が不安定な場合は復帰後に再度試してください。

Q. ボタンが反応しない / 表示が古い

Service Worker のキャッシュが原因の場合があります。ブラウザで「ハードリロード」（Ctrl+Shift+R / Cmd+Shift+R）を試してください。

15. データ構造（参考）

Firebase Realtime Database のツリー構造です。技術担当者向けの参考情報。

users/{uid}                プロフィール: name, email, role(staff|manager|office), color_index, created_at, updated_at
meta/firstOfficeUid        最初に登録した office の uid（不変、role validate 用）
employees/{empId}          社員マスタ: name, dept, workNo, empNo, active
punches/{pushId}           打刻ログ（追記専用）:
                             employeeId, employeeName, employeeDept,
                             type ∈ { in | out | break_start | break_end },
                             timestamp(ServerValue.TIMESTAMP),
                             dateKey "YYYY-MM-DD",
                             periodKey "YYYY-MM-H1|H2",
                             deviceUid, deviceLabel
editLogs/{pushId}          打刻修正履歴（manager/office のみ書込）:
                             targetPunchId, beforeValue, afterValue,
                             reason, editedBy, editedByEmail, editedAt
attendance/months/{YM}     社労士書類生成のキャッシュ: records, edits, filename, year, month
periods/{periodKey}        将来用（現在は未使用）
devices/{uid}              端末ラベル（任意）

権限ルール（要約）

  • users: 自分の doc は本人のみ書込、role 変更は office のみ（自分自身は不可）
  • meta/firstOfficeUid: 一度書き込まれたら不変
  • employees: 全員読込可、書込は office のみ
  • punches: 全員読込可、書込は本人 + サーバー時刻一致が必須（manager/office は時刻自由）
  • editLogs: manager/office のみ読み書き、追加のみ（更新・削除不可）
  • attendance/*: office のみ読み書き

サーバー時刻の改ざん耐性

一般ユーザー（staff）の打刻は timestamp === now を .validate ルールで検証しているため、過去・未来時刻の偽装は不可能です。/admin での手動修正は manager/office のみ可能で、必ず editLogs に before/after が記録されます。
